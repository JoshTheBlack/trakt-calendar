================================================================================
0. GOAL
================================================================================
Turn the app from single-user/no-auth into multi-user with authentication:

  - Login via username/password, Login with Plex, Login with Trakt — all three
    can be linked to the same account.
  - First-run onboarding creates the initial admin account.
  - New accounts (of any login method) require an admin-issued invite to
    register at all, and then require admin approval before they can use the
    app. Admin approval is split into two independent grants per user:
    calendar access and distrakt access.
  - An admin interface for issuing invites, approving/rejecting accounts,
    resetting passwords, revoking sessions, deleting accounts, and viewing
    linked identities.
  - The main calendar page's data is cached globally per (endpoint, 7-day
    window) in UTC; only the "not watching" decisions and view preferences are
    per-user. Each user can publish their own calendar as a read-only public
    page (no login, ZERO live API calls — stale cache is served as-is).
  - Distrakt remains an easter-egg-only route (not linked in the UI), but now
    additionally requires: the requesting user is logged in, Trakt-linked, and
    admin-approved for distrakt access. Distrakt reads Trakt data using the
    user's own linked Trakt token, not an app-wide token.
  - Calendar API calls get a caching rework: global UTC-normalized cache,
    refreshed on a TTL (Trakt's calendar endpoints don't support conditional
    requests — confirmed live, see §1.11), with network/language/month-boundary
    filtering done server-side against the cached raw data instead of
    re-fetching per filter combination. All times are stored UTC and rendered
    in the VIEWER's timezone (§1.15).
  - EVERY existing route gets an explicit authorization requirement (§8).
    This is the highest-priority item in the whole plan: today
    `GET /api/settings` returns the Trakt access token, Trakt client secret,
    TMDB key, and every *arr API key to any unauthenticated caller.

This is being built BEFORE the planned DRY/SRP/pythonic/security refactor of
the existing codebase, specifically so that refactor doesn't have to redo any
of this. Every design choice below favors patterns that survive that refactor
(clear module boundaries, no cross-cutting hacks, one obvious owner per piece
of state).


================================================================================
1. DECISIONS LOCKED IN (with rationale)
================================================================================

1.1  Storage backend: SQLite (stdlib `sqlite3`), not more JSON files.
     Why: multi-user data is inherently relational (users <-> linked
     identities <-> sessions <-> approvals <-> per-user calendar state <-> share
     links) and needs atomic writes under concurrent access. JSON-file writes
     from two simultaneous requests (e.g. two users registering, or an admin
     approving while a user changes their not-watching state) risk corruption
     or lost updates. SQLite is in the stdlib (no new dependency) and gives
     transactions for free. New file: `data/app.db` (alongside the existing
     `data/settings.json`, which stays as-is for app-wide config).
     The existing per-endpoint/year/month `state_*.json` files (app/state.py)
     get replaced by a `calendar_not_watching` table (§5.6) — part of chat G.

1.1a SQLite operational contract (set once in chat A, in ONE module —
     `app/db.py` — and never re-decided later):
       - `PRAGMA journal_mode=WAL` (concurrent readers during a write).
       - `PRAGMA foreign_keys=ON` on EVERY connection (SQLite defaults it off;
         it is per-connection, not per-database).
       - `PRAGMA busy_timeout=5000`.
       - `PRAGMA synchronous=NORMAL` (safe under WAL, much faster than FULL).
       - One connection per thread (`threading.local`), NOT one global
         connection. `check_same_thread` stays at its default True.
       - stdlib `sqlite3` is BLOCKING and every route in this app is
         `async def`. EVERY db call MUST go through
         `await anyio.to_thread.run_sync(...)`. `app/db.py` exposes async
         helpers (`fetch_one`, `fetch_all`, `execute`, `transaction`) that do
         this internally; no other module may `import sqlite3`.
       - Writes spanning more than one statement go in an explicit transaction
         helper, not autocommit.

1.1b Schema migrations: `app/db.py` owns a migration runner and a
     `schema_version` table (single row, integer). Migrations are an ordered
     list of `(version, sql_or_callable)` in ONE module, applied inside a
     transaction at startup, forward-only. Chats A, G, I, and J each APPEND to
     that list — they never edit an existing migration and never create their
     own runner. This is what keeps four separate chats from producing four
     incompatible schema-creation paths.

1.2  Password hashing: Argon2id via the `argon2-cffi` package (new dependency).
     Why: current OWASP-recommended default for new applications; memory-hard,
     tunable, resists GPU/ASIC cracking better than bcrypt at equivalent cost.
     Use `argon2.PasswordHasher()` library defaults — do NOT hand-tune.
     CRITICAL: hashing/verifying is deliberately 50-200ms of CPU. Called
     directly from an `async def` route it stalls the entire event loop and
     makes `/login` a trivial denial-of-service lever. Every `hash()` and
     `verify()` call MUST be wrapped in `await anyio.to_thread.run_sync(...)`.
     Also call `ph.check_needs_rehash()` on successful verify and transparently
     upgrade the stored hash when the library's defaults move.

1.3  Per-user "Login with Trakt": redirect (authorization_code) OAuth flow,
     NOT the device-code flow the app already uses for its one existing
     app-wide connection.
     Why: single-click login UX for end users (bounce to Trakt, bounce back)
     instead of "here's a code, go type it into this other page, then come
     back."
     One app-wide Trakt API app registration (client_id/secret, set once in
     Settings) is used for ALL users — each user's "Login with Trakt" click
     authorizes against that one registered app and Trakt hands back a
     per-user access/refresh token pair, which is what's stored per-user.
     Trakt requires one app registration per client_id, so per-user
     credentials were never realistic for a smooth login button.
     REDIRECT URI (LOCKED): `<public_base_url>/auth/trakt/callback`, e.g.
     `https://shows.example.com/auth/trakt/callback`. The admin registers
     exactly that on the Trakt API app and sets `public_base_url` in Settings
     to the matching origin (§1.16b). Trakt requires the `redirect_uri` sent in
     the code exchange to match the registered value EXACTLY — it is built from
     `public_base_url`, never from request headers.
     The EXISTING device-code flow (app/trakt_auth.py) is KEPT (§2 item 2) as
     an admin-only break-glass path in Settings.

1.4  Per-user "Login with Plex": PIN-based flow (matches Overseerr/Ombi-style
     "Sign in with Plex"), NOT a redirect/OAuth flow — Plex doesn't offer one.
     Flow: request a PIN from plex.tv -> open a Plex-hosted popup for the user
     to approve it -> poll plex.tv until the PIN is linked -> fetch the Plex
     account's identity (id, username, email, thumb) with the resulting token.
     IMPORTANT ASYMMETRY vs. Trakt: Plex requires NO admin-side app
     registration and NO redirect URI. The only things needed are an
     `X-Plex-Client-Identifier` (a UUID generated once on first startup and
     persisted in `app_meta` — identifies this app INSTALLATION to Plex, not
     any particular user) and a product-name header, both auto-generated with
     zero Settings UI. Do NOT build a "Plex client id/secret" settings field —
     there is nothing for the admin to configure here.
     NOTE: a Plex login proves only "this person controls SOME plex.tv
     account." It is not a membership check against any particular server; the
     intended users are not on this instance's Plex server. Registration
     gating is handled entirely by the invite system (§1.19).

1.5  Sessions: server-side, DB-backed, opaque cookie.
     The cookie holds only a random opaque session ID
     (`secrets.token_urlsafe(32)`); the actual session row (user_id,
     created_at, expires_at, last_seen_at, user_agent/ip for the admin's
     session list) lives in SQLite. Revoking a session (admin forces logout,
     password reset, etc.) is just deleting a row — instant, no waiting for a
     token to expire.
     Cookie flags: `HttpOnly`, `SameSite=Lax`, `Path=/`, no `Domain`
     attribute, and `Secure` per §1.16a.
     Cookie name: `__Host-tns_session` when Secure is on (the `__Host-` prefix
     is browser-enforced: requires Secure + Path=/ + no Domain, and blocks a
     sibling subdomain from overwriting it); plain `tns_session` when not.
     LIFETIME (LOCKED): 14-day sliding expiry, refreshed on use at most once
     per hour (so an active session doesn't cause a write on every request),
     with a 60-day ABSOLUTE cap from `created_at` that sliding cannot extend.
     There is no security argument for revision 1's 30 days, and a shorter
     window costs active users nothing because the expiry slides.
     Session rows are hard-deleted on: logout, password change, admin revoke,
     admin disable, account delete. Expired rows are swept by the existing
     heartbeat loop in app/main.py.

1.6  CSRF posture (HARDENED — revision 1's rationale was partly wrong).
     Revision 1 claimed a cross-site page can't trigger a same-origin JSON
     fetch without a CORS preflight. True for `application/json`, but the app's
     own `POST /api/settings` ALSO accepts `application/x-www-form-urlencoded`
     (it calls `request.form()`, and `python-multipart` is a dependency) — a
     form-encoded POST is a CORS "simple request" with NO preflight, so the
     only thing protecting it was `SameSite=Lax` alone, with no defense in
     depth and no protection from a hostile same-site subdomain.
     LOCKED POLICY, applied to EVERY mutating endpoint:
       (a) `SameSite=Lax` session cookie (as before).
       (b) Mutating endpoints accept `Content-Type: application/json` ONLY.
           Anything else -> 415. `POST /api/settings` drops its form branch;
           the Settings UI already posts JSON.
       (c) An Origin/Referer check: if `Origin` is present it MUST equal
           `public_base_url`'s origin, else 403. If absent, `Sec-Fetch-Site`
           must be `same-origin` or `none`. Implemented ONCE as middleware in
           chat B, not per-route.
       (d) The OAuth/PIN callbacks are top-level GET navigations, which
           `SameSite=Lax` DOES send cookies on — they get session-bound
           state/nonce protection instead (§4.2).
     No classic HTML `<form>` POST may be introduced anywhere. If one ever is,
     (b) and (c) must be revisited.

1.7  Registration: no email required. Username + password + (client-side and
     server-side) confirmation, plus a valid invite token (§1.19). Rationale:
     no alerting/notification system planned, and password resets are an
     admin-driven manual action (§6), so email has no functional use yet.
     Don't add an email column — adding it later, if a self-service reset is
     ever wanted, is a trivial migration; adding it now would be a field with
     no consumer.

1.8  Existing single-user data migration: on first startup after upgrade, if
     `data/settings.json` already has a valid `trakt_access_token`, treat it as
     already-linked to the bootstrap admin account created during onboarding
     (§3) — no forced re-auth. `trakt_client_id`/`trakt_client_secret` in
     settings.json become the one app-wide OAuth app's credentials (§1.3).
     Additionally (revision 2): the existing `data/state_*.json` files are
     imported into `calendar_not_watching` + `calendar_view_state` under the
     bootstrap admin's user_id (chat G). Distrakt's `data/distrakt/*.json` and
     `data/watch_state.json` are NOT auto-migrated — the production instance
     doesn't run distrakt, and chat J ships an explicit JSON export/import
     (§1.20) which covers the "move my dev data to the server" case and
     doubles as a per-user backup facility.

1.9  Public calendar share pages: server-rendered on request, reading ONLY from
     the SQLite per-user state + the global cache (§5) — never calling Trakt
     live, EVEN IF THE CACHE IS STALE. A public request never triggers a
     refetch and never extends the cache; it serves whatever is there and
     renders a "data as of <timestamp>" line. This is what makes "zero API
     calls" literally true and prevents an unauthenticated visitor from
     spending the instance's Trakt rate limit. If a requested month has no
     cached data at all, the page renders an empty month with that same as-of
     line rather than 500ing.

1.10 Share link format: three public route SHAPES, all resolving to the same
     page for the same user, each in its OWN URL namespace so they cannot
     collide:
       /s/<opaque-token>    — DEFAULT. Unguessable, revocable/regenerable
                              independently of anything else.
       /u/<username>        — human-readable, opt-in.
       /c/<custom-slug>     — user-chosen, opt-in, globally unique.
     Rationale for separate prefixes rather than one `/shared/<x>` segment:
     with a single namespace, one user's custom slug could shadow another
     user's username and the resolver would need a precedence rule invisible to
     both users. Separate prefixes make each lookup a single unambiguous query.
     A user can have all three active at once; the "Share" button copies
     whichever the user has set as `preferred_kind`. All three keep working
     regardless of which is preferred (switching preference does not invalidate
     the others). The point of the token/slug options is that a user whose
     username here differs from their username elsewhere may not want the
     username form handed out — it is a choice about which link they COPY, not
     a secrecy guarantee about the others.
     UNIQUENESS (LOCKED): `custom_slug` and `username` are validated against
     EACH OTHER in BOTH directions — a slug may not equal any existing
     username, and a username may not equal any existing slug — so the three
     forms stay mutually consistent even though they live in separate
     namespaces. Both are case-insensitive (`COLLATE NOCASE`), both match
     `^[a-z0-9][a-z0-9_-]{1,31}$`, and both are checked against a reserved
     list: admin, administrator, api, auth, static, healthz, login, logout,
     register, onboarding, settings, distrakt, shared, s, u, c, pick, me, new,
     null, undefined, root, system, support, help.
     RETIREMENT: when an account is deleted (§6), its username, slug, and token
     are recorded in `retired_identifiers` (§5.15) and are blocked from reuse BY
     DEFAULT. Otherwise a new user could claim a deleted user's username and
     silently inherit every `/u/<username>` link already shared in the wild.
     The admin CAN deliberately release a retired identifier from the admin UI
     (§6) — the safe behavior is the default, and the "this person is coming
     back and wants their name" case is one explicit click rather than a
     silent-by-accident hazard. Releasing a `token` is never offered: share
     tokens are random and there is no legitimate reason to reissue a specific
     one. Only usernames and slugs can be released.

1.11 Calendar refresh gating: TTL-based, not conditional-request-based.
     LIVE-TESTED (using this instance's real Trakt credentials) on
     `GET /calendars/all/shows/premieres/<date>/7`: the response carries no
     `ETag` and no `Last-Modified` header (full header dump inspected
     directly, not just the CORS expose-list). Trakt's calendar endpoints are
     public schedule data, not tied to a user's watch history, so
     `/sync/last_activities` doesn't apply here either (that only reflects a
     user's own watch actions, used separately for watched-progress/distrakt
     history — see §5.2). Conclusion: cache with a TTL, default 10 minutes,
     admin-tunable in Settings (mirroring today's `cache_ttl_minutes` pattern).
     A shorter TTL naturally suits the current/near month where premiere dates
     shift more; a long-past or far-future month rarely needs a refetch at all
     before a user even looks at it again — no special-casing needed beyond the
     TTL check itself.

1.12 SQLite migration scope beyond the auth tables — audited the rest of the
     app's JSON-file persistence (app/cache.py, app/state.py, app/distrakt.py,
     app/watch_history.py, app/config.py, app/logos.py, app/perftrace.py) so we
     don't end up running two persistence mechanisms side by side for the same
     kind of problem. Decisions:
       - `app/cache.py` (generic sha1-keyed TTL blob cache for show
         detail/cast/episode lookups) gets folded into ONE generic cache table
         (`api_cache`, §5.7) that both it and the new calendar cache use. There
         is ONE TTL-blob-cache mechanism in the codebase, not two. Revision 1
         left "bespoke calendar_cache table vs. rows in api_cache" to the
         implementer; it is now LOCKED: one table, `api_cache`, with structured
         cache keys. There is NO `calendar_cache` table. Scope: chat G.
       - `app/distrakt.py` (per-month JSON docs) and `app/watch_history.py`
         (single watch_state.json) both move to SQLite AND become per-user
         (§1.13) — not just a storage-format change but a scope change, so this
         is its own chat (J), not folded into G.
       - `app/config.py` (settings.json) stays as it is — a single JSON file,
         app-wide only. See §1.14.
       - `app/logos.py`'s cached logo images/SVGs stay on the filesystem —
         they're binary/text static-asset caches served as files, not state;
         SQLite would just mean the same bytes as a BLOB with no concurrency or
         correctness benefit.
       - `app/perftrace.py` doesn't persist anything (logging only).

1.13 Distrakt becomes fully per-user, not a shared/global tracker.
     Each distrakt-approved user tracks their own independent set of shows,
     builds their own Cleanup/Keepup/Completed/Abandoned buckets from their own
     Trakt watch history, and generates their own pair of Discord posts.
     Different users are very likely watching substantially different shows, so
     there is no meaningful "the roster" to share. Every table in §5.8-§5.11 is
     keyed by `user_id` accordingly. The `/distrakt` page, once reached via the
     easter egg and gated (chat K), shows the logged-in user's own tracker,
     full stop.
     App-wide visual config that genuinely IS shared across all users' Discord
     posts (network_emojis, default_network_emoji, tmdb_api_key — all already
     in settings.json) stays global; do not add per-user overrides of those.

1.14 settings.json boundary rule: it holds ONLY app-wide/global configuration —
     never anything specific to an individual user. Rationale: keeping it a
     single flat file (rather than folding it into SQLite "for consistency")
     means an admin can still hand-edit or blank out settings.json to recover a
     container that fails to start, without needing sqlite3 tooling. Any config
     specific to a user MUST live in SQLite instead. HARD RULE.
       APP-WIDE (stays in settings.json): Trakt OAuth app credentials, app-wide
       Trakt token, `public_base_url`, `cookie_secure`, `trusted_proxy_ips`,
       `allow_open_registration`, calendar cache TTL, cache size cap,
       Sonarr/Radarr/Seerr config, TMDB key, network emoji map, and the
       DEFAULT values of every per-user pref below.
       PER-USER (moves to SQLite, §5.1 / §5.1a): timezone, endpoint,
       card_style, day_packing, hide_not_watching, network_filter, genres,
       countries.
     Each per-user field's settings.json counterpart is now purely the SEED
     used when a user row is created — changing it later affects new users, not
     existing ones. This applies to `timezone` (§1.15) and, per the §1.17e
     measurement, to `genres`/`countries` as well: they are no longer sent to
     Trakt at all, so there is no cache-multiplication reason to keep them
     app-wide, and making them per-user is what lets one user include JP/KR
     shows while another excludes them from the very same cached data.

1.15 ALL TIMES ARE STORED UTC AND RENDERED IN THE VIEWER'S TIMEZONE.
     Today `app/trakt.normalize()` converts to `settings.timezone` at FETCH
     time, baking an app-wide timezone into every derived field (`air_date`,
     `air_ts`, `air_display`, `air_time`, `day_of_week`) and into the
     month-boundary trim. That cannot be shared across users.
     NEW CONTRACT:
       - The cache (§5.7 / §1.17) stores Trakt's response with times
         UNCONVERTED (raw ISO-UTC strings, exactly as Trakt sent them).
       - `normalize()` becomes a pure read-time function taking an explicit
         `tz` argument. No module-level or settings-derived timezone.
       - Each user has a `timezone` (users.timezone, §5.1), SEEDED FROM
         settings.json's app-wide default (currently America/New_York), NOT
         auto-detected from the browser. A timezone picker appears on the
         calendar page (reusing the existing `app/timezones.py` grouped option
         builder), defaulting to the user's saved value, persisting on change,
         with a one-click "use my device timezone" button next to it that fills
         in `Intl.DateTimeFormat().resolvedOptions().timeZone`.
         WHY NOT AUTO-DETECT: US TV schedules are published in Eastern, and
         "Tuesday night TV" is a real thing users reason about. Silently
         auto-detecting shifts shows across day boundaries for anyone on a VPN,
         travelling, or simply in Europe/Asia, with no visible explanation for
         why a show moved a day. The operator's configured default is a
         deliberate choice and is the better starting point; users who want
         their own local rendering are one click away.
         This is a DISPLAY choice only — because storage is UTC (above), both
         behaviors are available at zero cost and switching is instant. If a
         genuine "always group by US Eastern regardless of my clock" mode is
         ever wanted, it is just a saved timezone of America/New_York, which
         the default already gives you.
       - CONSEQUENCE — THE MONTH BOUNDARY IS VIEWER-DEPENDENT. An item airing
         `2026-03-01T02:00Z` is 28 Feb in America/Los_Angeles and 1 Mar in
         Europe/Athens. The read path therefore loads the cached windows
         covering [month_start - 1 day, month_end + 1 day] in UTC and trims to
         the month IN THE VIEWER'S TZ after conversion. NEVER trim in UTC.
       - Share pages have no logged-in viewer: they render in the OWNER's saved
         timezone by default, overridable with a `?tz=` query param (validated
         against the IANA database; invalid -> owner's default, never an error).

1.16 REVERSE PROXY / HTTPS (this instance runs behind Traefik, which terminates
     TLS and enforces the HTTPS route).
     The app itself is served by Hypercorn over plain HTTP inside the
     container, so `request.scheme` is `"http"` unless forwarded headers are
     both sent AND trusted. Hypercorn's `forwarded_allow_ips` defaults to
     `127.0.0.1`, and Traefik connects from another container IP — so revision
     1's "detect via request scheme" would have SILENTLY SHIPPED SESSION
     COOKIES WITHOUT THE `Secure` FLAG on every real deployment.
     LOCKED HANDLING, three independent parts:
       (a) COOKIE SECURE FLAG — a `cookie_secure` setting with values
           `always` (DEFAULT) | `never` | `auto`. Default `always` means the
           correct production behavior does not depend on proxy configuration
           at all. `never` exists for plain-HTTP LAN use; `auto` falls back to
           scheme detection for anyone who wants it. README documents that
           `never` is required if you genuinely serve over HTTP, and that any
           HTTPS deployment (Traefik, nginx proxy manager, Caddy, ...) leaves
           it at `always`.
       (b) ABSOLUTE URLS — every absolute URL the app generates (the Trakt
           `redirect_uri`, share links shown in the UI, the link embedded in
           distrakt's Discord post) is built from the `public_base_url`
           setting. NEVER from the `Host` header or `request.url`. This makes
           the app structurally immune to Host-header injection and removes any
           dependency on proxy config for URL correctness. `public_base_url` is
           validated on save (absolute http(s) URL, no trailing slash, no path)
           and is REQUIRED before "Login with Trakt" can be enabled.
       (c) CLIENT IP — needed for login rate limiting (§1.18) and the admin's
           session list. `X-Forwarded-For` is honored ONLY when the immediate
           peer is a trusted proxy.
           CONFIGURED IN TWO PLACES, deliberately:
             - `trusted_proxy_ips` in Settings (admin-editable, comma-separated
               CIDRs) is the value the APP uses for its own header parsing.
               Exposed in the admin UI because the correct value depends on the
               operator's docker network and changes when they restructure it —
               that shouldn't require an env-var edit and a container restart.
               The Settings field shows the CURRENTLY DETECTED peer IP next to
               the input, so the admin can see what to enter instead of
               guessing their Docker subnet.
             - `TRUSTED_PROXY_IPS` env var (default `127.0.0.1/32`) remains,
               and is the SEED for the setting on first run plus the value
               passed to hypercorn's `--forwarded-allow-ips` in the Dockerfile
               CMD — hypercorn reads its flag at process start and cannot be
               reconfigured from the running app, so that half stays an env
               var. README documents both and gives a Traefik snippet.
           If the peer is not trusted, the peer IP is used and forwarded
           headers are ignored entirely.
           SECURITY NOTE: too permissive lets an attacker spoof their IP and
           evade rate limiting; too restrictive collapses every user to the
           proxy's IP and makes rate limiting global. Chat A logs a startup
           WARNING when `TRUSTED_PROXY_IPS` is at its default while forwarded
           headers are actually present, since that combination is almost
           always a misconfiguration.

1.17 CALENDAR CACHE SHAPE (replaces revision 1 §5.5, which was not viable).
     Revision 1 said "cache the full normalized item list, pre-filter, and do
     all filtering at read time." Two problems: (1) `genres` and `countries`
     are Trakt QUERY PARAMETERS in `app/trakt._build_url`, so today's response
     is ALREADY filtered server-side by Trakt — going pre-filter means fetching
     unfiltered, and the default filter is aggressive (9 excluded genres, a
     36-country allowlist); for the `shows` endpoint ("every episode of every
     show airing"), an unfiltered month with `extended=full,images` is a very
     large payload. (2) "normalized" and "pre-filter" contradict each other,
     and normalizing bakes in the timezone (§1.15).
     LOCKED DESIGN:
       (a) FETCH IN 7-DAY WINDOWS, NOT MONTHS. Trakt's calendar path is
           `/calendars/all/<path>/<start_date>/<days>` — the date range IS the
           natural pagination unit, and the §1.11 live test already used 7.
           A month view = 5-6 window fetches, each cached and TTL'd
           independently, each of bounded size. This also makes §1.15's
           "month ±1 day" trivial: load the windows overlapping the requested
           UTC range. Windows are aligned to FIXED 7-day boundaries from a
           fixed epoch (NOT relative to "today") so two users looking at the
           same month hit the same cache rows.
       (b) STORE RAW, PRUNED, UTC. Store Trakt's response entries as-is (raw
           ISO-UTC timestamps, no conversion, no `Item` normalization), PRUNED
           to only the fields the app actually consumes: `normalize()`'s inputs
           plus the fields the filters read. `extended=full` returns a lot the
           app never touches; pruning is the single biggest size lever and it
           also stops the cache from growing when Trakt adds fields. Pruning
           happens once at write time. Chat G verifies correctness by diffing
           the pruner's allowlist against every `media.get(...)` /
           `entry.get(...)` in `normalize()`.
       (c) CACHE KEY: `calendar:<endpoint_key>:<window_start_date>`
           Nothing else. Per (e) the cached data is complete and unfiltered, so
           there is no filter dimension to key on. `genres`, `countries`,
           `network_filter`, and `language` are ALL per-user read-time filters
           applied to the cached blob — which is the actual win revision 1 was
           after, now achieved completely rather than partially.
       (d) PAGINATION — MEASURED, NOT ASSUMED. Live-tested 2026-07-21 against
           `/calendars/all/shows/2026-07-01/7?extended=full,images` (348 items):
           sending `X-Pagination-Page: 1` + `X-Pagination-Limit: 300` returns
           the SAME 348 items as sending no pagination headers at all, and the
           response carries NO `X-Pagination-*` headers whatsoever. Calendar
           endpoints are genuinely unpaginated and ignore the headers the app
           currently sends. CONCLUSION: no page loop is needed, and there is no
           latent truncation bug in today's code.
           Chat G does TWO cheap things instead of building an unused loop:
           stop sending the pagination headers on calendar calls (they are
           meaningless there), and log a WARNING if an `x-pagination-page-count`
           header ever appears on a calendar response — zero-cost
           future-proofing if Trakt ever changes this.
           Also do NOT do a two-phase "get ids, then fetch each show" pass —
           that trades one large response for hundreds of small ones, which is
           strictly worse against Trakt's rate limit and much slower.
       (e) SIZE — MEASURED 2026-07-21, DECISION MADE. Revision 1 worried the
           unfiltered `shows` endpoint ("every episode of every show airing")
           would be too large to cache. It is not. Real numbers, one 7-day
           window, unfiltered, pruned per (b), zlib level 6:

             window        endpoint          items   pruned      gzipped
             2026-07-01    shows               348    271.7 KB    70.1 KB
             2026-07-01    shows/premieres      42     35.8 KB    12.0 KB
             2026-07-01    shows/new            21     18.5 KB     6.4 KB
             2026-07-01    shows/finales        13      9.6 KB     3.5 KB
             2026-07-01    movies                9      5.5 KB     2.2 KB
             2025-09-22    shows               566    433.1 KB   111.5 KB   <- worst case
             2025-09-22    shows/premieres      71     54.3 KB    17.1 KB
             2025-09-22    shows/new            18     13.5 KB     4.7 KB

           2025-09-22 is a PAST premiere week with a fully-populated schedule
           and is the realistic ceiling: ~111 KB gzipped for the heaviest
           7-day `shows` window. A full heavy month of `shows` is therefore
           ~550 KB gzipped; ALL FIVE endpoints for a heavy month ~700 KB; a
           full year of all five ~8 MB. That is nothing.
           NOTE ALSO: near-future months are SPARSE and fill in over time
           (2026-09/10 returned 50-129 items per window vs. 566 for the same
           calendar week in 2025) because Trakt hasn't populated them yet.
           That is exactly the behavior the TTL exists to pick up, and it is
           an argument for re-fetching near-future months rather than trusting
           an early empty-ish result forever.

           DECISION (LOCKED): NO FILTERING IS APPLIED TO CACHED CALENDAR DATA.
           `genres` and `countries` are NO LONGER sent as Trakt query params on
           calendar fetches. Every endpoint caches the complete unfiltered
           worldwide result, and ALL filtering — genres, countries, networks,
           language — happens at READ TIME against the cached blob, per user.
           Rationale beyond size: the whole point is that one user wanting
           JP/KR shows and another excluding them must both be servable from
           the same cache. Filtering at fetch time makes that structurally
           impossible, and the measurements show it buys nothing.
           CONSEQUENCE for §1.14 and §1.17c: `genres` and `countries` move OUT
           of the cache key (nothing left to key on) and OUT of app-wide-only
           status — they become PER-USER read-time filters alongside
           `network_filter`, stored in `user_prefs` (§5.1a), with the
           settings.json values retained purely as the DEFAULT seeded into each
           new user's prefs.
       (g) READ-TIME FILTER PREDICATE — SPECIFIED AND VERIFIED, NOT INVENTED.
           The one real risk in (e) was that a Python reimplementation of
           Trakt's `genres`/`countries` filter would disagree with what Trakt
           was doing server-side, silently changing what users see. That was
           tested live on 2026-07-21 and the predicate below reproduced Trakt's
           own filtering EXACTLY — 8 of 8 windows, ~2000 items, zero
           false-drops and zero false-keeps, across BOTH filter styles.
           Chat G implements exactly this; do not redesign it:

             parse(spec): split on ',', strip, lowercase; a leading '-' puts
               the bare token in EXCLUDES, otherwise it goes in INCLUDES.
             keep(media):
               genres  = {lowercased strings in media["genres"] or []}
               country = lowercase(media["country"] or "")
               if G_EXC and (genres & G_EXC):        -> drop
               if G_INC and not (genres & G_INC):    -> drop
               if C_EXC and country in C_EXC:        -> drop
               if C_INC and country not in C_INC:    -> drop
               otherwise                             -> keep

           VERIFIED AGAINST both filter styles the app actually produces:
             - exclude-style (this instance's live settings:
               `-anime,-home-and-garden,-music,-talk-show` / `-kr`) — 4/4 MATCH
             - include-style countries (config.py's DEFAULT_COUNTRIES 36-country
               allowlist + DEFAULT_GENRES exclusions, i.e. what every FRESH
               INSTALL gets) — 4/4 MATCH
             Sample: `shows` 2025-09-22, 566 unfiltered -> Trakt returned 426
             with the live filter and 272 with the defaults; the predicate
             produced 426 and 272 respectively, item-for-item identical.

           TWO NOTES for the implementer:
             - Filter on the RAW genre slugs from the cached blob. Do NOT filter
               after `normalize()`, which rewrites genres as
               `g.replace("-", " ").title()` ("game-show" -> "Game Show") and
               would break every hyphenated match.
             - Missing/empty fields: an item with no `genres` is KEPT under an
               exclude-only spec (empty set intersects nothing) and DROPPED
               under an include spec. That matched Trakt's behavior, but the
               live sample contained only 1 item with no genres and 0 with no
               country, so the evidence is thin — chat G should keep explicit
               unit tests for the empty-field cases rather than relying on the
               integration fixture to cover them.
       (f) BLOB STORAGE: pruned JSON, zlib-compressed, in a BLOB column.
           Two-line change, large payoff on highly-repetitive JSON; do it from
           the start rather than retrofitting.

1.18 Login rate limiting: a `login_attempts` TABLE (§5.13), not an in-memory
     window. Survives restarts, and the admin UI can surface current lockouts.
     LOCKED POLICY: track per (username, client_ip) independently. Lock after 5
     failed attempts within 15 minutes, for 15 minutes, counted per-username
     AND per-IP separately — so one attacker can't lock out every user by
     burning attempts, and one victim IP can't be used to spray many usernames.
     Failures and lockouts return the SAME generic "invalid credentials"
     response and the same timing class as an ordinary failure — a lockout must
     not be an oracle for "this username exists." Successful login clears that
     username's counter. Old rows swept by the heartbeat loop.
     The same table also rate-limits: registration attempts per IP (10/hour),
     invite redemption per IP (10/hour), and public share-page requests per IP
     (120/minute — generous, purely anti-scrape, since those never touch Trakt).

1.19 REGISTRATION IS INVITE-GATED (new in revision 2).
     A Plex or Trakt login proves only that someone controls SOME account on
     that service, and the intended users are not on this instance's Plex
     server, so no membership check is available. Without gating, anyone on the
     internet can auto-register and sit in the approval queue forever.
     LOCKED DESIGN:
       - `invites` table (§5.12): token, created_by, label, created_at,
         expires_at, max_uses, used_count, revoked, plus
         `grants_calendar_on_accept` (bool).
       - Registration in ANY form — username/password, first-time Plex login,
         first-time Trakt login — requires a valid, unexpired, unrevoked,
         under-quota invite. The invite is presented as
         `<public_base_url>/register?invite=<token>` and is carried through the
         OAuth/PIN handshake in the SERVER-SIDE state record (§4.2, §5.14), NOT
         in a cookie or the redirect URL, so it cannot be tampered with
         mid-flow.
       - `grants_calendar_on_accept` DEFAULTS TO TRUE. An invite is already a
         deliberate act of trust by the admin — making the invitee then wait in
         an approval queue is friction with no added safety. They register and
         land straight on the calendar. The admin can uncheck it per-invite for
         a "let them in the door but I'll look first" case, and the
         awaiting-approval page (§3) still exists for that path and for
         `allow_open_registration` instances.
       - It NEVER grants distrakt access — distrakt is always a deliberate,
         separate, manual admin grant AFTER registration (§3), because distrakt
         exposes a user's private watch history and is the one place where an
         over-broad grant is a data-exposure bug rather than a convenience.
         There is deliberately no `grants_distrakt_on_accept` field; do not add
         one.
       - `allow_open_registration` (settings.json, DEFAULT False) bypasses the
         invite requirement when an operator wants a fully open instance.
         Approval gating (§3) still applies when it is on.
       - An invite that is expired / revoked / exhausted / never-existed
         returns the SAME "this invite link is not valid" page in all four
         cases.

1.20 DISTRAKT JSON EXPORT / IMPORT (new in revision 2).
     Chat J ships `GET /api/distrakt/export` (the requesting user's complete
     distrakt dataset — every month, every show row, watch state, progress,
     movie watches — as one JSON document) and `POST /api/distrakt/restore`
     (the inverse). Purposes: move a dev machine's accurate test data onto the
     server after an upgrade, and give every user a real local backup that
     turns a data-loss event into a 30-second restore instead of a 15-minute
     rebuild.
     RULES:
       - Scoped to the REQUESTING user only. There is no cross-user import; the
         restoring user's id comes from the session and any `user_id` present
         in the file is IGNORED, not honored.
       - The document carries a `schema` version integer. Restore refuses a
         version it doesn't understand rather than guessing.
       - Restore is REPLACE, not merge, and runs in ONE transaction: delete the
         user's existing rows in the four distrakt tables, insert the file's. A
         merge would need conflict rules for every field and has no clear use
         case. The UI confirms with a typed acknowledgement.
       - NAMING: the existing `POST /api/distrakt/import` is "import premieres
         from the calendar" and is unrelated. Do NOT overload it. The new
         routes are `/api/distrakt/export` and `/api/distrakt/restore`.
       - The export contains no tokens and no other user's data.

================================================================================
3. ACCOUNT / APPROVAL STATE MACHINE
================================================================================
Every user row has:
  - `is_admin` (bool) — exactly one bootstrap admin created at onboarding; more
    can be promoted later via the admin UI (§2 item 9).
  - `calendar_approved` (bool, default False except for the bootstrap admin and
    invitees redeeming a `grants_calendar_on_accept` invite)
  - `distrakt_approved` (bool, default False ALWAYS — never granted
    automatically by any invite or any login method)
  - `is_disabled` (bool) — admin can deactivate an account without deleting it.

Login outcomes:
  - NO ACCOUNT EXISTS AT ALL (fresh instance) -> force `/onboarding` for every
    request until the first admin is created (mirrors the existing
    `settings.configured` gate pattern in app/main.py's `/` route, just at a
    higher priority — checked before that one).
    THE BOOTSTRAP ADMIN MUST SET A USERNAME AND PASSWORD. The first admin may
    not be OAuth-only: if the sole way into the instance were "Login with
    Trakt," a Trakt outage, a revoked app registration, or a changed redirect
    URI would lock the operator out of their own instance with no recovery
    path. They can link Plex/Trakt afterwards from their profile.
    RACE: onboarding creation runs inside a single transaction that re-checks
    `SELECT COUNT(*) FROM users` and relies on a partial unique index
    guaranteeing at most one row with `is_bootstrap=1`, so two concurrent POSTs
    cannot both succeed. Cheap, so it is required.

3.1  UPGRADING THE EXISTING SINGLE-USER INSTANCE (the migration path)
     Today's instance has real data and NO user/password — `users` is empty on
     first startup after the upgrade, so the §3 gate forces `/onboarding` for
     every request. That IS the migration trigger; there is no separate
     migration mode. The onboarding form is the one place the operator sets the
     credentials they never had, and it does the adoption in one transaction:
       1. Create the bootstrap admin from the submitted username + password,
          with is_admin / is_bootstrap / calendar_approved / distrakt_approved
          all TRUE.
       2. Seed `user_prefs` (§5.1a) from settings.json's current values, so the
          calendar looks EXACTLY as it does today (same endpoint, card style,
          genres, countries, network filter) and `users.timezone` from
          settings.json's `timezone` (currently America/New_York).
       3. If settings.json has a `trakt_access_token`, adopt it as a
          `linked_identities` row (provider='trakt') for this admin. The
          immutable numeric id required by §4.3 is NOT in settings.json, so
          onboarding calls `GET /users/me?extended=full` with that token to
          fetch `ids.trakt` and the display name. If that call fails (expired
          token, revoked app, no network), create the admin WITHOUT the link
          and surface a "reconnect your Trakt account" notice in Settings —
          never block onboarding on a third-party call, and never store an
          identity row with a guessed id.
       4. Import the existing `data/state_*.json` files into
          `calendar_not_watching` + `calendar_view_state` under this admin's
          user_id (§1.8). Chat G owns the importer; chat C calls it. If chat G
          hasn't landed yet when chat C runs, chat C leaves a clearly-marked
          hook and chat G fills it — the files are not deleted until the import
          has run successfully at least once.
     The app-wide Trakt token in settings.json is NOT removed by this — it stays
     as the credential for app-wide calendar fetches (§5.2), independent of the
     admin's personal link.
  - USERNAME/PASSWORD, UNKNOWN USERNAME -> generic "invalid credentials" (don't
    leak which part was wrong). Verify against a dummy Argon2 hash when the
    username doesn't exist so the timing matches a real failure.
  - USERNAME/PASSWORD, CORRECT, `calendar_approved=False` -> a static "your
    account is awaiting admin approval" page. Same page is what an unapproved
    user hits if they try to reach the calendar directly.
  - PLEX/TRAKT LOGIN, IDENTITY NOT YET LINKED TO ANY ACCOUNT -> THIS IS A
    REGISTRATION. It requires a valid invite in the handshake's server-side
    state (§1.19), else the callback lands on the "this invite link is not
    valid" page and NO account is created. With a valid invite: auto-register a
    new account under that identity (no username/password set), landing on the
    awaiting-approval page unless the invite carried
    `grants_calendar_on_accept`.
  - PLEX/TRAKT LOGIN, IDENTITY ALREADY LINKED -> normal login as that account,
    subject to its existing approval state and `is_disabled`.
  - LINKING A SECOND METHOD to an already-logged-in account -> must be done
    while authenticated, from the profile screen, never from the logged-out
    login page. The link handshake is bound to the session (§4.2). If the
    incoming identity is ALREADY linked to a DIFFERENT account, the link is
    REFUSED with "that Plex/Trakt account is already linked to another user" —
    it is never silently moved.
  - A user may NOT unlink their LAST remaining login method if they have no
    password set (that would orphan the account). The admin can still unlink
    anything from the admin UI, and is warned when doing so orphans an account.
  - `is_disabled=True` -> login always fails with the generic message, all
    existing sessions are deleted, and the user's public share pages 404.
  - DISTRAKT ACCESS is checked independently: a `calendar_approved=True` user
    with `distrakt_approved=False` can use the calendar fully but /distrakt
    (even after finding the easter egg) shows "distrakt access not yet
    approved," and separately requires the user to actually have a linked Trakt
    identity (distrakt reads their personal Trakt history — a user who only
    ever logged in with username/password or Plex has nothing for distrakt to
    read until they link Trakt too).

4. SECURITY REQUIREMENTS (normative — carry into every chat)
================================================================================

4.1  GENERAL
  - Never log tokens, passwords, session IDs, or invite tokens. Add a logging
    filter that redacts them defensively; do not rely on discipline alone.
  - Argon2id via `argon2.PasswordHasher()` defaults, OFF-THREAD (§1.2).
  - Store Trakt/Plex tokens the same way the app already stores the app-wide
    Trakt token today (plain in the local SQLite file, consistent with
    settings.json already holding a plaintext token) — no new encryption scheme
    is being introduced here; the trust boundary is "whoever has filesystem
    access to this instance," unchanged from today. `data/app.db` is created
    with mode 0600.
  - Session IDs, share tokens, invite tokens, and OAuth state values all come
    from `secrets.token_urlsafe(32)`. Compare with `secrets.compare_digest`,
    never `==`.
  - `GET /api/settings` MUST NOT return secret values. It returns every
    non-secret field plus a `{field: bool}` map of which secrets are set.
    Secrets are WRITE-ONLY: the UI posts a new value, or omits the field to
    leave it unchanged. This is the single highest-value fix in the plan —
    today that route hands the Trakt access token, Trakt client secret, TMDB
    key, and all *arr API keys to any unauthenticated caller.
  - All new mutating routes require an active session except: login, register,
    onboarding-create-first-admin (only reachable when zero users exist), and
    the OAuth/PIN callback endpoints themselves. §8 is the complete matrix and
    covers existing routes too.
  - Public `/s/`, `/u/`, `/c/` routes are read-only in every sense: no write
    endpoints reachable, no session required or consulted, no live API calls
    (§1.9), and a miss returns an IDENTICAL 404 page regardless of whether the
    identifier never existed, belongs to a disabled user, or was retired.

4.2  HANDSHAKE BINDING (OAuth state / Plex PIN) — ACCOUNT-TAKEOVER CLASS
     An unbound callback is an account-takeover vector, not just a CSRF nit: if
     an attacker can get a victim's browser to complete a callback carrying the
     ATTACKER's Trakt or Plex identity while the victim is logged in, that
     identity becomes linked to the victim's account and the attacker can log
     in as the victim indefinitely. `SameSite=Lax` does NOT help here — these
     callbacks are top-level GET navigations, which Lax explicitly permits.
     LOCKED REQUIREMENTS for BOTH chat D and chat E:
       - An `auth_handshakes` table (§5.14): id, state (random, unique),
         provider, purpose ('login' | 'link'), session_id (nullable — set ONLY
         for purpose='link'), invite_token (nullable — set only for
         registration), pkce_verifier (nullable), plex_pin_id (nullable),
         created_at, expires_at (10 minutes), consumed_at.
       - The callback MUST: find the row by `state`; verify it is unconsumed
         and unexpired; verify that for purpose='link' the CURRENT request's
         session id equals the row's `session_id`; mark it consumed INSIDE THE
         SAME TRANSACTION (single-use enforced by the DB, not by a read-then-
         write); and only then exchange the code / read the PIN.
       - A missing, unknown, expired, or already-consumed state -> hard 400
         with a generic message. NEVER fall back to "no state, assume login."
       - Same rules for Plex: the PIN id is stored in the handshake row, and
         the poll endpoint refuses to look up a PIN that isn't bound to the
         caller's handshake.
       - Use PKCE on the Trakt flow if Trakt supports it; if it doesn't, record
         that in §2.1 and rely on the state binding alone.

4.3  IDENTITY KEYS — IDENTITY-REUSE CLASS
     `linked_identities.provider_user_id` MUST be the provider's IMMUTABLE
     NUMERIC ID:
       - Trakt: `ids.trakt` (int) from `GET /users/me?extended=full`.
       - Plex: the numeric `id` from the Plex account resource.
     NEVER the username, slug, or email. Trakt usernames and slugs are
     user-changeable and can be released and re-registered by someone else —
     keying on one means a released username silently inherits the linked
     account. `display_name` is stored separately, for DISPLAY ONLY, refreshed
     on each login; nothing may key off it.

4.4  ENUMERATION AND ERROR PARITY
  - Login failure, locked-out, disabled account, and unknown username all
    return the same message and status.
  - Invalid / expired / revoked / exhausted / never-existed invite all return
    the same page.
  - Share-link 404s are identical for all causes (§4.1).
  - Registration inherently reveals "username taken" — accepted, and the only
    enumeration oracle in the design. Rate-limited per §1.18.

4.5  AUTHORIZATION IS DENY-BY-DEFAULT
     Chat B introduces a mechanism requiring every route to DECLARE its access
     level. A route with no declaration is DENIED, and the app logs an ERROR at
     startup listing any undeclared routes (with a test that fails on any).
     This makes "someone adds a route and forgets to gate it" a loud startup
     error instead of a silent hole.

5. DATA MODEL (SQLite — `data/app.db`)
================================================================================
Shape, not final DDL — the implementing chat writes the real migration — but
this is the binding contract between chats. All timestamps are UTC; pick one
representation (ISO-8601 text or unix int) in chat A and use it everywhere.

5.1  `users`
       id INTEGER PK, username TEXT UNIQUE COLLATE NOCASE (nullable — OAuth-only
       accounts may never set one), password_hash TEXT (nullable),
       password_changed_at, is_admin, is_bootstrap, calendar_approved,
       distrakt_approved, is_disabled, timezone TEXT (nullable -> app default),
       created_at, updated_at, last_login_at.
       - `COLLATE NOCASE` on username is REQUIRED, else `Admin` and `admin`
         coexist as separate accounts.
       - `password_changed_at` exists so "log out everywhere on password
         change/reset" is enforceable rather than aspirational.
       - Partial unique index on `is_bootstrap=1` (§3 race guard).

5.1a `user_prefs` — per-user view preferences (§1.14 split)
       user_id (FK, UNIQUE), endpoint, card_style, day_packing,
       hide_not_watching, network_filter_json, genres, countries.
       Seeded from settings.json's values when a user is created, so existing
       behavior is unchanged for the bootstrap admin. Kept as its own table
       rather than columns on `users` so the auth model and the view model stay
       separable for the later refactor.
       `genres`/`countries` keep the exact same string format Trakt used as
       query params (`-anime,-music` style, leading `-` = exclude) so the
       existing Settings UI widget and the existing values migrate verbatim;
       chat G reimplements that string's semantics as a read-time predicate
       over `media.genres` / `media.country` instead of sending it to Trakt.

5.2  `linked_identities`
       id, user_id (FK ON DELETE CASCADE), provider ('plex' | 'trakt'),
       provider_user_id (IMMUTABLE NUMERIC ID — §4.3), display_name,
       access_token, refresh_token, token_expires_at, refreshing_until,
       created_at, last_login_at.
       UNIQUE (provider, provider_user_id) — this is what makes "already linked
       -> log in as that account" possible.
       A user's personal Trakt token here is what distrakt (chat J) uses per
       request, and what `fetch_last_activities` / `fetch_history` /
       `fetch_show_progress_detail` (app/trakt.py) get pointed at instead of
       the app-wide settings token, for that user's own distrakt/watch-history
       calls specifically. The app-wide CALENDAR fetches (§1.17) keep using the
       app-wide settings.json token, since calendar data isn't user-specific —
       and keeping them separate means calendar browsing keeps working for
       everyone even if one user revokes their personal Trakt authorization.
       Token refresh is SERIALIZED per identity row (`refreshing_until` +
       transaction) so two concurrent requests can't both spend the same
       single-use refresh token and invalidate each other.

5.3  `sessions`
       id TEXT PK (opaque token, the cookie value), user_id (FK ON DELETE
       CASCADE), created_at, expires_at, absolute_expires_at, last_seen_at,
       user_agent, ip_address. Index on (user_id), index on (expires_at).
       `ip_address` is personal data: it exists for the admin session list and
       is deleted with the session row; no separate audit log retains it.

5.4  `share_links`
       user_id (FK ON DELETE CASCADE, UNIQUE), token TEXT UNIQUE,
       custom_slug TEXT UNIQUE COLLATE NOCASE (nullable),
       preferred_kind ('token' | 'username' | 'slug'), enabled_token,
       enabled_username, enabled_slug (bools — a user may publish some forms
       and not others), created_at, token_rotated_at, plus the OWNER DEFAULTS
       for the §7.3 view options: endpoint, card_style, day_packing,
       hide_not_watching, network_filter_json, timezone.
       Query params override these per §2 item 5.

5.5  (RESERVED — revision 1's `calendar_cache` table DOES NOT EXIST. Calendar
     caching lives in `api_cache`, §5.7, per §1.12 and §1.17.)

5.6  `calendar_not_watching`  (replaces `app/state.py`'s `state_*.json` files)
       user_id (FK ON DELETE CASCADE), endpoint, year, month, item_id (the
       calendar card's `data-id` = slug preferred, else str(trakt_id)),
       created_at. PRIMARY KEY (user_id, endpoint, year, month, item_id).
       One row per "marked not watching" item, per user.

5.6a `calendar_view_state`  (the per-viewer change-detection fields — §2 item 6)
       user_id (FK ON DELETE CASCADE), endpoint, year, month, last_count,
       last_show_ids_json, history_json, updated_at.
       PRIMARY KEY (user_id, endpoint, year, month).

5.7  `api_cache`  (replaces `app/cache.py`'s hashed-file directory AND carries
     the calendar cache — ONE TTL-blob-cache mechanism, §1.12)
       cache_key TEXT PK, payload BLOB (zlib-compressed JSON), cached_at,
       ttl_seconds (nullable — per-entry override, since season lookups use a
       24h TTL and calendar windows use ~10 min), byte_size.
       App-wide, not user-scoped — detail/cast/episode lookups aren't
       user-specific any more than calendar premieres are.
       KEY CONVENTIONS: existing detail lookups keep app/cache.py's current key
       (the full request URL), so nothing about their behavior changes beyond
       where the bytes live; calendar entries use the structured key in §1.17c.
       `app/cache.py` KEEPS its `get(key, ttl)` / `set(key, value)` signature so
       every existing call site in app/trakt.py is untouched — only the
       module's internals change. The old `data/cache/*.json` directory is
       deleted at startup once, after the first successful migration run.
       WHAT ACTUALLY GOES IN HERE (measured 2026-07-21 on the live instance):
         - Existing detail/cast/episode lookups, today `data/cache/*.json`:
           11 MB across 53 files, i.e. ~213 KB PER ENTRY. These are whole
           `extended=full` show/people/season responses and they are by far the
           bigger consumer — one entry per (show) + (show,people) +
           (show,season), so a user browsing 200 shows across a few seasons
           each can reach ~1000 entries / ~210 MB on their own. Multi-user
           multiplies the variety of shows looked at, not the per-entry size.
         - Calendar windows, new: ~111 KB gzipped for the worst 7-day `shows`
           window, ~700 KB for a heavy month across all five endpoints, ~8 MB
           for a full year of all five (§1.17e). Effectively a rounding error
           next to the detail lookups.
       EVICTION: the heartbeat loop deletes entries past their TTL plus a grace
       period, and enforces a total-size cap — DEFAULT 1 GB (not 256 MB; the
       213 KB/entry measurement above makes 256 MB roughly 1200 detail lookups,
       which a handful of active users would blow through), admin-tunable in
       Settings, evicting least-recently-used first. `byte_size` is stored per
       row so the cap can be enforced with a single SUM query rather than
       stat-ing the database file.
       FUTURE (not in scope, noted so it isn't re-derived): the detail-lookup
       payloads are unpruned whole responses. Applying the same §1.17b pruning
       to them would likely cut that 213 KB/entry by 60-70%. Deliberately out
       of scope for chat G because it changes `fetch_details`' contract; revisit
       during the DRY/SRP refactor.

5.8  `distrakt_shows`  (replaces `app/distrakt.py`'s per-month JSON docs —
     PER-USER per §1.13)
       user_id (FK ON DELETE CASCADE), month ('YYYY-MM'), trakt_id, tmdb, slug,
       media, title, season, network, abandoned, abandoned_form, watched,
       total, cadence, premiere, finale, bucket, started_airing,
       finished_airing.
       UNIQUE (user_id, month, trakt_id, season) — mirrors `_find_show`'s
       lookup key in the current module, scoped per user and per month.

5.8a `distrakt_months`  (the month-level fields — §2 item 8)

6. ADMIN INTERFACE SCOPE (v1)
================================================================================
  - List all accounts: username (or linked-identity display name if no
    username), linked providers, calendar_approved / distrakt_approved /
    is_disabled state, created_at, last login, last session activity.
  - Approve/reject calendar access, approve/reject distrakt access (independent
    toggles, per §0).
  - Promote/demote admin (§2 item 9). GUARD: the last remaining admin cannot be
    demoted, disabled, or deleted.
  - Issue invites: label, expiry, max uses, `grants_calendar_on_accept`;
    list/copy/revoke existing invites; see redemptions (§5.12a).
  - Reset a user's password (admin sets a temporary one directly — no email
    flow, per §1.7). Resetting stamps `password_changed_at` and deletes ALL of
    that user's sessions.
  - Disable / re-enable an account. Disabling deletes all sessions and 404s the
    user's share pages.
  - Revoke a specific session, or "log out everywhere" for a user.
  - View which identities (Plex/Trakt) are linked to an account, and unlink one
    if needed (e.g. account recovery after a user loses Plex access), with a
    warning if the unlink would leave the account with no way to log in.
  - TWO DISTINCT DESTRUCTIVE ACTIONS (§2 item 10). They solve different
    problems and both are wanted; do not collapse them into one:
      (i) WIPE DATA — keeps the account, its identities, its username/slug, and
          its share links; deletes the user's calendar not-watching rows, view
          state, and all four distrakt tables; sets `is_disabled=1` and revokes
          sessions. This is the "start this person over" / "they're not using
          it, reclaim the rows, but I might re-enable them" action. Reversible
          by re-enabling; nothing is retired.
     (ii) DELETE ACCOUNT — full hard delete. RULES:
          * Requires typing the account's username to confirm.
          * Runs in ONE transaction. `ON DELETE CASCADE` on every FK to `users`
            handles the fan-out; chat F asserts in a test that deleting a user
            leaves ZERO orphan rows in every child table.
          * Records the account's username, custom_slug, and share token in
            `retired_identifiers` (§5.15) so they are blocked from reuse by
            default — else a new user could claim a deleted user's username and
            silently inherit every `/u/<username>` link already shared.
          * Cannot delete the last remaining admin.
          * Cannot delete YOURSELF (avoids the "locked out of my own instance"
            footgun; demote-then-delete from another admin account works).
  - RETIRED IDENTIFIERS panel: lists retired usernames/slugs with the date, and
    a "release" action per row that deletes the block so the identifier becomes
    claimable again — for when a deleted user re-registers and wants their name
    back. The confirmation states plainly that any previously shared
    `/u/<name>` or `/c/<slug>` link will then point at whoever claims it next.
    Tokens are never releasable (§1.10).
  - Explicitly OUT of v1 scope: bulk actions, an audit log, email anything.

7. CALENDAR PAGE CHANGES (gated + shared)
================================================================================

7.1  The existing `/` route (app/main.py) gains an auth requirement:
     `calendar_approved` session required, else redirect to
     login/awaiting-approval as appropriate.

7.2  AUTHENTICATED READ PATH:
       1. Determine the viewer's timezone (users.timezone -> app default).
       2. Compute the UTC window range covering
          [local month start - 1 day, local month end + 1 day] (§1.15).
       3. For each aligned 7-day window in that range: read `api_cache`; if
          missing or past TTL, fetch from Trakt (with the §1.17d page loop),
          prune, compress, store.
       4. Concatenate; `normalize()` each entry with the VIEWER's tz.
       5. Trim to the viewer's LOCAL month.
       6. Apply per-user read-time filters (network_filter, language).
       7. Overlay the viewer's `calendar_not_watching` rows.
       8. Apply the viewer's hide_not_watching / card_style / day_packing.
     Two different logged-in users looking at the same month see the same shows
     (possibly with different day groupings if their timezones differ), but
     their own not-watching marks and their own view preferences.

7.3  PUBLIC `/s/<token>`, `/u/<username>`, `/c/<slug>` variants do the SAME read
     path with these differences: no session consulted; step 3's FETCH BRANCH
     IS DISABLED (§1.9 — serve cache or serve empty, never fetch); no write
     affordances rendered; a "data as of <timestamp>" line is shown.
     View options resolve as: query param -> owner's `share_links` default ->
     app default. SUPPORTED PARAMS (LOCKED):
       `year`, `month`, `endpoint`, `card` (vertical|horizontal|poster),
       `packing` (stacked|packed), `hidenw` (0|1), `networks` (comma list),
       `tz` (IANA name).
     Every param is validated against a whitelist; anything invalid falls back
     silently to the next tier rather than erroring.

7.4  `POST /api/state` becomes a DELTA endpoint: `{item_id, not_watching: bool}`
     for a single item, plus a separate `{last_count, last_show_ids}` write for
     the change-detection fields. Today app/static/js/app.js sends the ENTIRE
     `notWatching` array on every toggle (a read-modify-write), which loses
     updates when the same user has two tabs open. Now that the storage is rows
     rather than one JSON document, a delta is the natural shape and the
     lost-update class disappears. The JS toggle handler changes accordingly.

7.5  A TIMEZONE PICKER appears in the calendar header (using the existing
     grouped options from app/timezones.py), defaulting to the user's saved
     timezone, persisting on change, with a "use my device timezone" button
     beside it (§1.15). No automatic browser detection.

7.6  EASTER-EGG BEHAVIOR WHEN DISTRAKT ISN'T APPROVED
     Today both unlocks in app/static/js/app.js call the same `revealSecret()`:
     the Konami sequence (10 keys) and 7 rapid clicks on the footer
     `.version-tag`. Both set `localStorage.distraktRevealed` and navigate to
     `/distrakt`, and the flag makes a permanent `#distraktNav` button appear.
     Separately, the Konami code ON the distrakt page plays
     `/static/audio/distrakt.mp3` (app/static/js/distrakt.js:558).
     NEW BEHAVIOR:
       - The calendar template renders `window.DISTRAKT_AVAILABLE` server-side
         from the session's real gate result (distrakt_approved AND a linked
         Trakt identity). No probe endpoint — a client-side check that asks the
         server "am I allowed?" is itself a disclosure, and the template
         already has the session.
       - AVAILABLE -> today's behavior, unchanged: set the localStorage flag,
         navigate, and surface the nav button on later visits.
       - NOT AVAILABLE -> the KONAMI code plays `/static/audio/distrakt.mp3`
         and does nothing else: no navigation, no localStorage flag, no nav
         button, no message. It reads as a self-contained easter egg with no
         hint that a gated feature exists behind it.
       - NOT AVAILABLE -> the 7-tap VERSION CLICK does NOTHING AT ALL. It is a
         silent tactile unlock with no feedback channel; firing audio from an
         accidental footer tap is startling and would out the easter egg to
         someone who wasn't looking for it. Audio stays on the Konami code,
         which nobody enters by accident.
       - `window.DISTRAKT_AVAILABLE` is UX ONLY. The server gates /distrakt and
         every /api/distrakt/* route independently (§8, chat K); a tampered
         flag gains the client nothing.

    made them calendar-level). Sonarr/Radarr writes go into the operator's real
    shared libraries, and Seerr's integration posts through a single app-wide
    API key, so every user's request would be attributed to the admin — which
    both misattributes and bypasses the approval workflow Seerr exists to
    provide. Ordinary users who want something requested use Seerr directly,
    where they already have their own accounts.
    UI CONSEQUENCE: the add-to-library / request buttons and the integration
    health chips are hidden entirely for non-admins — not rendered-then-403'd,
    since a button that always fails is worse than no button. The calendar
    template gates them on the same `is_admin` flag it already has in context.
    PER-USER *arr/SEERR ENDPOINTS — EVALUATED AND REJECTED (do not relitigate;
    if it is ever revisited, start from these constraints rather than
    rediscovering them):
      (i) SERVER-SIDE per-user config (a `user_integrations` table; the app
          calls the user's own Sonarr URL) is easy to build and creates a
          textbook SSRF hole. The user supplies a URL that OUR SERVER fetches
          with OUR network position — which reaches every other container on
          the Docker network, the Traefik dashboard, `127.0.0.1`, and cloud
          metadata endpoints, none of which are reachable from the internet.
          Success/failure alone turns the feature into an internal port
          scanner. Doing it safely means: deny-by-default on loopback/private/
          link-local ranges with an explicit admin allowlist, re-validating the
          resolved IP AFTER DNS resolution (against rebinding), disabling
          redirect following, hard timeouts, and never returning the upstream
          body to the caller. That is a real project, not a settings field.
          It ALSO doesn't solve the actual case: if a user's Sonarr is on THEIR
          LAN behind THEIR router, this server cannot reach it at all — the
          feature only works for users who expose their services publicly.
     (ii) BROWSER-SIDE fetch from the user's own machine is the architecturally
          correct topology (their browser IS on their LAN) and is dead on
          arrival in modern browsers: our page is HTTPS behind Traefik, so a
          `fetch()` to `http://192.168.x.x:8989` is blocked as MIXED CONTENT
          with no override; Sonarr/Radarr send no `Access-Control-Allow-Origin`
          for our origin, so CORS blocks reading the response; Chrome's Private
          Network Access additionally requires a public->private preflight that
          Sonarr won't answer; and the user's API key would live in page JS,
          readable by any XSS. Making it work requires the user to put a real
          cert and public DNS on their LAN service, at which point (i) applies
          anyway.
    CONCLUSION: the supported path for non-admin users is Seerr with their own
    Seerr accounts, which already exists, already has an approval workflow, and
    is exactly the problem Seerr solves. Do not build scaffolding for per-user
    integrations now.
